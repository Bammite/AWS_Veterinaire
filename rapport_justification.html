Partie A — Analyse de l'application

L'analyse structurelle ci-dessous sert de fondement aux décisions d'architecture d'infrastructure prises sur AWS.

1. Découpage en tiers et nature du composant à exécuter

L'application suit un modèle à 3-tiers (présentation Thymeleaf, métier Spring Boot, données SGBD). Bien que séparés logiquement, la présentation et le métier sont packagés dans un seul livrable monolithique (JAR exécutable) exécuté dans des conteneurs ou machines virtuelles via un script de démarrage Systemd standardisé.

2. État applicatif : stateless vs stateful

Le serveur d'application Spring Boot est entièrement stateless (sans état), tandis que la base de données est stateful. L'application ne stockant aucun état de session ou fichier localement sur le disque, toute donnée est externalisée dans le SGBD relationnel pour permettre un passage à l'échelle horizontal sans perte de données.

3. Persistance : choix de MySQL et conséquences RDS

Le projet supportant nativement MySQL et PostgreSQL via des profils Spring Boot, nous avons choisi MySQL associé à Amazon RDS for MySQL Multi-AZ. Ce choix élimine l'administration système du SGBD et offre une réplication synchrone automatique, tout en nécessitant l'injection de variables d'environnement pour configurer l'URL JDBC de production.

4. Disponibilité, montée en charge et tolérance aux pannes

L'architecture cible une haute disponibilité en s'étendant sur deux zones de la région us-east-1. Elle couple un Application Load Balancer (ALB) frontal pour distribuer les requêtes et isoler les pannes à un Auto Scaling Group (ASG) applicatif privé et un déploiement RDS Multi-AZ avec basculement automatique.

5. Gestion de la configuration et des secrets

Les mots de passe d'administration de la base de données ne doivent pas figurer dans le code source ni dans l'IaC. Ils sont générés aléatoirement et stockés de façon sécurisée dans AWS Secrets Manager, puis récupérés dynamiquement au démarrage de l'instance via son Rôle IAM d'Instance Profile.

6. Surface de sécurité : exposition réseau et chiffrement

La surface d'attaque est réduite en isolant les serveurs de calcul et de données dans des sous-réseaux privés, seuls les ALB restant publics. Les communications d'administration se font via AWS Systems Manager (Session Manager) sans ports ouverts sur Internet (SSH désactivé), et toutes les données sont chiffrées au repos (KMS) et en transit (TLS).

Note Pédagogique
Ce découpage logique en 3-tiers s'intègre rigoureusement dans la configuration réseau physique AWS mise en œuvre, garantissant un cloisonnement strict par groupes de sécurité chaînés (Internet → ALB → EC2 → RDS).

2. Argumentaire du Choix Retenu & Compromis Assumés

L'Option 1 (EC2 + ASG + ALB) a été retenue comme étant la plus adaptée pour ce projet d'étude en raison de deux facteurs essentiels :

  • Intérêt Pédagogique Majeur : Contrairement à App Runner qui marque totalement le réseau, cette architecture nous oblige à concevoir et configurer chaque composant d'infrastructure (les tables de routage, les NAT Gateways, les groupes de sécurité en chaîne). C'est le choix optimal pour valider concrètement les compétences requises pour la certification AWS SysOps/Solutions Architect.
  • Zéro Coût résiduel (FinOps) : Le déploiement s'intègre à 100% dans le AWS Free Tier grâce aux instances t3.micro pour les instances EC2, nous évitant des coûts inattendus lors de la validation du TP.

Compromis assumés :

  • La surcharge d'exploitation (Operations) : Nous acceptons de gérer le système d'exploitation Ubuntu, l'installation de Java et l'écriture du script de boot user_data.sh.
  • Le temps de scaling : La mise en service d'une nouvelle instance en cas de montée en charge prend environ 90 secondes (le temps d'allouer la VM et de lancer l'application), ce qui est supérieur aux conteneurs serverless mais tolérable pour ce cas d'usage.

Questions Clés pour la Défense Orale (Soutenance)

Voici les réponses préparées pour répondre aux interrogations classiques du jury d'évaluation :

Q1. Que se passe-t-il si une zone de disponibilité (AZ) subit une panne totale ?
L'ALB détecte immédiatement l'indisponibilité de l'instance dans l'AZ en panne via les contrôles de santé. Il coupe le trafic vers celle-ci et le redirige à 100 % vers l'autre instance EC2 saine située dans la seconde AZ. Parallèlement, l'instance de base de données RDS bascule automatiquement en synchrone vers la réplique secondaire située dans la zone saine, sans perte de données.
Q2. Où est stocké le mot de passe de la base de données RDS et comment y accède-t-on de manière sécurisée ?
Le mot de passe est généré de manière aléatoire et non visible par Terraform, puis stocké directement dans AWS Secrets Manager (chiffré via KMS). Les instances applicatives possèdent un Rôle IAM d'Instance (Instance Profile) les autorisant à interroger Secrets Manager au démarrage pour injecter le mot de passe en mémoire vive. Aucun secret n'est codé en dur dans le code ou l'IaC.
Q3. Comment l'application gère-t-elle la montée en charge automatique ?
Notre ASG est associé à des alarmes Amazon CloudWatch. Si la charge processeur moyenne (CPU) du cluster dépasse le seuil de 70% pendant 2 minutes consécutives, l'alarme commande à l'ASG d'instancier une nouvelle machine virtuelle privée. À l'inverse, si l'usage du CPU retombe sous 30% pendant 5 minutes, une règle de scale-in détruit proprement les instances excédentaires.
Q4. Quel est le coût estimé en production réelle et comment l'optimiser ?
Hors Free Tier, cette architecture coûte environ $134/mois (2 EC2, 1 RDS Multi-AZ, 2 NAT Gateways, 1 ALB). Pour un environnement réel, on pourrait : (1) désactiver le Multi-AZ de RDS en non-production, (2) utiliser une seule NAT Gateway partagée pour les tests, (3) utiliser des instances EC2 Spot pour l'application, et (4) opter pour des processeurs Graviton AWS (t4g) 20% plus performants et moins chers.
